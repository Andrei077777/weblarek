# Интернет-магазин «Web-Larёk»

«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектура приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component

Является базовым классом для всех компонентов интерфейса.
Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`constructor(container: HTMLElement)` - принимает ссылку на DOM элемент за отображение, которого он отвечает.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные, которые необходимо отобразить в интерфейсе, записывает эти данные в поля класса и возвращает ссылку на DOM-элемент. Предполагается, что в классах, которые будут наследоваться от `Component` будут реализованы сеттеры для полей с данными, которые будут вызываться в момент вызова `render` и записывать данные в необходимые DOM элементы.  
`setText(element: HTMLElement, value: unknown): void` - защищённый утилитарный метод для установки текстового содержимого элемента.  
`setDisabled(element: HTMLElement, state: boolean): void` - защищённый утилитарный метод для блокировки/разблокировки элемента (атрибут `disabled`).  
`toggleClass(element: HTMLElement, className: string, force?: boolean): void` - защищённый утилитарный метод для переключения класса элемента.  
`setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>`

#### Класс Api

Содержит в себе базовую логику отправки запросов.

Конструктор:  
`constructor(baseUrl: string, options: RequestInit = {})` - В конструктор передается базовый адрес сервера и опциональный объект с заголовками запросов.

Поля класса:  
`baseUrl: string` - базовый адрес сервера  
`options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get(uri: string): Promise<object>` - выполняет GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер  
`post(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<object>` - принимает объект с данными, которые будут переданы в JSON в теле запроса, и отправляет эти данные на ендпоинт переданный как параметр при вызове метода. По умолчанию выполняется `POST` запрос, но метод запроса может быть переопределен заданием третьего параметра при вызове.  
`handleResponse(response: Response): Promise<object>` - защищенный метод проверяющий ответ сервера на корректность и возвращающий объект с данными полученный от сервера или отклоненный промис, в случае некорректных данных.

#### Класс EventEmitter

Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<string | RegExp, Set<Function>>)` - хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков, которые будут вызваны при срабатывании события.

Методы класса:  
`on<T extends object>(event: EventName, callback: (data: T) => void): void` - подписка на событие, принимает название события и функцию обработчик.  
`emit<T extends object>(event: string, data?: T): void` - инициализация события. При вызове события в метод передается название события и объект с данными, который будет использован как аргумент для вызова обработчика.  
`trigger<T extends object>(event: string, context?: Partial<T>): (data: T) => void` - возвращает функцию, при вызове которой инициализируется требуемое в параметрах событие с передачей в него данных из второго параметра.


### Типы данных

В этом разделе описываются все интерфейсы и типы, используемые для обмена данными в приложении (файл `src/types/index.ts`).

**`IProduct`** — интерфейс, описывающий структуру одного товара:

- `id: string` — уникальный идентификатор товара;
- `title: string` — название;
- `description: string` — описание;
- `image: string` — путь к изображению;
- `category: string` — категория;
- `price: number | null` — цена (`null`, если товар бесценный/недоступен для покупки).

**`IBuyer`** — интерфейс, описывающий данные покупателя:

- `payment: TPayment | null` — выбранный способ оплаты (`null`, пока не выбран);
- `address: string` — адрес доставки;
- `email: string` — электронная почта;
- `phone: string` — телефон.

**`TPayment`** — тип-перечисление для способа оплаты. Значения: `'card' | 'cash'`.

**`TBuyerErrors`** — тип для результата валидации данных покупателя. Описывается как `Partial<Record<keyof IBuyer, string>>`: для каждого невалидного поля содержит строку с текстом ошибки; валидные поля в объекте отсутствуют.

**`ApiPostMethods`** — тип-перечисление для HTTP-методов, используемых в запросах с телом. Значения: `'POST' | 'PUT' | 'DELETE'`.

**`IApi`** — интерфейс, описывающий контракт базового API-клиента:

- `get<T extends object>(uri: string): Promise<T>`;
- `post<T extends object>(uri: string, data: object, method?: ApiPostMethods): Promise<T>`.

**`ApiProductsResponse`** — тип ответа сервера с каталогом товаров:

- `total: number` — общее количество товаров;
- `items: IProduct[]` — массив товаров.

**`OrderRequestData`** — тип данных, отправляемых на сервер при оформлении заказа:

- `buyer: IBuyer` — данные покупателя (во второй части проекта будет дополнен списком товаров).

**`OrderConfirmation`** — тип ответа сервера, подтверждающего заказ:

- `message: string` — сообщение об успешном оформлении;
- `totalSum: number` — итоговая сумма заказа.

#### Типы слоя представления

Эти типы добавлены во второй части проекта. Они построены на базе уже объявленных типов и не изменяют их — описывают данные, которые Презентер передаёт компонентам представления в метод `render`.

- **`IPage`** — данные главной страницы: `counter: number` (счётчик корзины) и `catalog: HTMLElement[]` (готовые карточки каталога).
- **`ICardCatalog`** — данные карточки каталога: `id`, `title`, `category`, `image`, `price`.
- **`ICardPreview`** (расширяет `ICardCatalog`) — данные карточки просмотра: добавляет `description`, текст кнопки `button` и флаг `buttonDisabled`.
- **`ICardBasket`** — данные строки корзины: `id`, `index`, `title`, `price`.
- **`IBasketView`** — данные корзины: `items: HTMLElement[]` и `total: number`.
- **`IFormState`** — общее состояние формы: `valid: boolean` и `errors: string`.
- **`IOrderForm`** — данные формы первого шага: `payment`, `address`.
- **`IContactsForm`** — данные формы второго шага: `email`, `phone`.
- **`ISuccess`** — данные окна успеха: `total: number`.
- **`IOrderRequest`** — полезная нагрузка запроса оформления заказа: `IBuyer & { items: string[]; total: number }`.

### Слой коммуникации

В этом разделе описывается класс, ответственный за взаимодействие с сервером.

#### Класс ApiService

Назначение: выполнение сетевых запросов к API сервера «веб-ларёк». Использует композицию — принимает в конструкторе объект, реализующий интерфейс `IApi` (экземпляр класса `Api`), и использует его методы `get`/`post`.

Конструктор:
`constructor(api: IApi)` — принимает API-клиент, который будет использоваться для запросов.

Методы:

- `getProducts(): Promise<ApiProductsResponse>` — выполняет GET-запрос на эндпоинт `/product/` и возвращает объект с массивом товаров (`{ total, items }`).
- `sendOrder(orderData: IOrderRequest): Promise<OrderConfirmation>` — выполняет POST-запрос на эндпоинт `/order/`, передавая данные покупателя вместе со списком id товаров и итоговой суммой, и возвращает объект с подтверждением заказа.

### Модели данных

В этом разделе описаны классы, отвечающие за хранение и управление состоянием приложения. Они не зависят от сетевого слоя и слоя представления — только хранят данные и предоставляют методы доступа к ним.

Каждая модель получает в конструкторе брокер событий (`IEvents`) и при любом изменении хранимых данных эмитит соответствующее событие. Сами модели на события не подписываются и логику приложения не содержат — за это отвечает Презентер.

#### Класс Products

Назначение: хранение каталога товаров и товара, выбранного для детального просмотра.

Конструктор:
`constructor(events: IEvents)` — принимает брокер событий для уведомления об изменениях.

Поля:
`items: IProduct[]` — массив всех товаров каталога;
`selectedProduct: IProduct | null` — товар, выбранный для подробного отображения.

Методы:

- `setItems(items: IProduct[]): void` — сохраняет массив товаров в модель и эмитит `catalog:changed`.
- `getItems(): IProduct[]` — возвращает массив всех товаров.
- `getItemById(id: string): IProduct | undefined` — возвращает товар по его id.
- `setSelectedProduct(product: IProduct): void` — сохраняет товар для детального просмотра и эмитит `preview:changed`.
- `getSelectedProduct(): IProduct | null` — возвращает выбранный товар или `null`.

#### Класс Cart

Назначение: хранение списка товаров, добавленных пользователем в корзину.

Конструктор:
`constructor(events: IEvents)` — принимает брокер событий для уведомления об изменениях.

Поля:
`items: IProduct[]` — массив товаров, выбранных для покупки.

Методы:

- `addItem(product: IProduct): void` — добавляет товар в корзину (если его там ещё нет) и эмитит `cart:changed`.
- `removeItem(id: string): void` — удаляет товар из корзины по id и эмитит `cart:changed`.
- `clear(): void` — очищает корзину и эмитит `cart:changed`.
- `getItems(): IProduct[]` — возвращает массив товаров в корзине.
- `getTotalPrice(): number` — возвращает общую стоимость товаров в корзине.
- `getCount(): number` — возвращает количество товаров в корзине.
- `isInCart(id: string): boolean` — проверяет, есть ли товар с указанным id в корзине.

#### Класс Buyer

Назначение: хранение данных покупателя (способ оплаты, адрес, контакты).

Конструктор:
`constructor(events: IEvents)` — принимает брокер событий для уведомления об изменениях.

Поля:
`data: IBuyer` — объект с данными покупателя (`payment`, `email`, `phone`, `address`).

Методы:

- `setPayment(payment: TPayment): void`, `setEmail(email: string): void`, `setPhone(phone: string): void`, `setAddress(address: string): void` — устанавливают отдельные поля данных покупателя; каждый эмитит `buyer:changed`.
- `getData(): IBuyer` — возвращает объект со всеми данными покупателя.
- `clear(): void` — сбрасывает все данные покупателя к начальным значениям и эмитит `buyer:changed`.
- `validate(): TBuyerErrors` — проверяет корректность введённых данных (поле валидно, если не пустое) и возвращает объект с ошибками по полям.

### Слой представления (View)

Все компоненты представления наследуются от базового класса `Component<T>` и работают по единым правилам:

- разметку компонент получает клонированием соответствующего `<template>` из `index.html` (утилита `cloneTemplate`);
- все нужные DOM-элементы находятся в конструкторе и сохраняются в полях класса;
- все слушатели событий устанавливаются один раз в конструкторе;
- компонент не хранит данные приложения и не содержит логики — он только отображает переданные данные и сообщает о действиях пользователя через брокер событий;
- метод `render(data)` записывает данные в поля (через сеттеры) и возвращает корневой DOM-элемент.

#### Класс Gallery и HeaderView

Назначение: главная страница. Отвечает за каталог товаров (`<main class="gallery">`) и кнопку корзины со счётчиком в шапке.

`constructor(container: HTMLElement, events: IEvents)`

Сеттеры: `counter: number` — число товаров в корзине; `catalog: HTMLElement[]` — список карточек каталога.

События: по клику на кнопку корзины эмитит `basket:open`.

#### Класс Modal

Назначение: модальное окно. Самостоятельный компонент — от него не наследуются другие классы, в него можно поместить разметку любого компонента. Видимость переключается модификатором `modal_active`.

`constructor(container: HTMLElement, events: IEvents)`

Сеттеры: `content: HTMLElement` — содержимое окна.  
Методы: `open()` — открыть окно; `close()` — закрыть окно (очищает содержимое).

События: по клику на крестик или вне окна закрывается и эмитит `modal:close`.

#### Класс Card (абстрактный)

Назначение: общий родитель для трёх карточек товара. Выносит общий функционал: хранение `id` (нужен только для событий) и вывод названия и цены.

`constructor(container: HTMLElement, events: IEvents)`

Сеттеры: `id: string`, `title: string`, `price: number | null` (выводит «Бесценно», если цена `null`).

#### Класс CardCatalog (наследник Card)

Назначение: карточка товара в каталоге (темплейт `#card-catalog`).

Сеттеры: добавляет `category` (с подстановкой модификатора фона из `categoryMap`) и `image`.

События: по клику на карточку эмитит `card:select` с `{ id }`.

#### Класс CardPreview (наследник Card)

Назначение: карточка товара в детальном просмотре (темплейт `#card-preview`).

Сеттеры: добавляет `category`, `image`, `description`, `button` (текст кнопки) и `buttonDisabled`.

События: по клику на кнопку эмитит `card:add` с `{ id }` (решение «добавить или удалить» принимает Презентер по состоянию корзины).

#### Класс CardBasket (наследник Card)

Назначение: строка товара в корзине (темплейт `#card-basket`).

Сеттеры: добавляет `index` (порядковый номер).

События: по клику на кнопку удаления эмитит `card:remove` с `{ id }`.

#### Класс Basket

Назначение: корзина (темплейт `#basket`). Список товаров, итоговая стоимость и кнопка оформления.

`constructor(container: HTMLElement, events: IEvents)`

Сеттеры: `items: HTMLElement[]` — строки товаров (если пусто — выводит «Корзина пуста» и блокирует кнопку оформления); `total: number` — итоговая стоимость.

События: по клику на кнопку «Оформить» эмитит `order:open`.

#### Класс Form (абстрактный)

Назначение: общий родитель для двух форм оформления. Выносит общий функционал: обработку ввода и отправки формы, управление кнопкой отправки и выводом ошибок. Имена событий строятся от атрибута `name` формы.

`constructor(container: HTMLFormElement, events: IEvents)`

Сеттеры: `valid: boolean` — активность кнопки отправки; `errors: string` — текст ошибок.

События: при вводе в любое поле эмитит `<name>:change` с `{ field, value }`; при отправке — `<name>:submit`.

#### Класс OrderForm (наследник Form)

Назначение: форма первого шага (темплейт `#order`) — выбор оплаты и адрес.

Сеттеры: `payment` (подсвечивает выбранную кнопку модификатором `button_alt-active`), `address`.

События: `order:change` (в т.ч. при выборе способа оплаты), `order:submit`.

#### Класс ContactsForm (наследник Form)

Назначение: форма второго шага (темплейт `#contacts`) — почта и телефон.

Сеттеры: `email`, `phone`.

События: `contacts:change`, `contacts:submit`.

#### Класс Success

Назначение: окно успешного оформления заказа (темплейт `#success`).

`constructor(container: HTMLElement, events: IEvents)`

Сеттеры: `total: number` — списанная сумма.

События: по клику на кнопку эмитит `success:close`.

### События приложения

Все события вынесены в объект `Events` (`src/utils/constants.ts`).

События Моделей данных:

| Событие | Когда генерируется |
| --- | --- |
| `catalog:changed` | сохранён массив товаров каталога (`Products.setItems`) |
| `preview:changed` | выбран товар для просмотра (`Products.setSelectedProduct`) |
| `cart:changed` | изменилось содержимое корзины (`Cart.addItem/removeItem/clear`) |
| `buyer:changed` | изменились данные покупателя (`Buyer.setPayment/setEmail/setPhone/setAddress/clear`) |

События Представлений:

| Событие | Когда генерируется | Данные |
| --- | --- | --- |
| `card:select` | клик по карточке каталога | `{ id }` |
| `card:add` | клик по кнопке покупки в карточке просмотра | `{ id }` |
| `card:remove` | клик по кнопке удаления товара в корзине | `{ id }` |
| `basket:open` | клик по кнопке корзины | — |
| `order:open` | клик по кнопке «Оформить» в корзине | — |
| `order:change` | изменение поля формы первого шага | `{ field, value }` |
| `order:submit` | клик по кнопке «Далее» | — |
| `contacts:change` | изменение поля формы второго шага | `{ field, value }` |

### Презентер

Презентер не выделен в отдельный класс и реализован в основном скрипте приложения `src/main.ts` — на странице одна логическая область, поэтому одного презентера достаточно. Презентер использует инверсию зависимостей: создаёт брокер событий, модели данных и компоненты представления и связывает их между собой.

Презентер состоит исключительно из обработчиков событий и сам события не генерирует. Он подписывается на события моделей и представлений: в ответ на действия пользователя вызывает методы моделей, а в ответ на изменение данных в моделях — перерисовывает представления. Перерисовка происходит только при обработке события об изменении данных модели или при открытии модального окна. Подготовка данных для компонентов (например, сборка полного URL изображения или вычисление текста кнопки покупки) тоже выполняется в презентере.
